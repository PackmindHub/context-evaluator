# Evaluation Prompt Optimization Plan

## Context

Analysis of evaluation `093eedea` (Supabase, 26 files, unified mode) revealed that **66% of the $7.72 cost** was spent on 12 evaluators that found zero issues. The 5 optimizations below target prompt size, evaluator selection, and output efficiency to reduce cost by an estimated 30-50% per evaluation.

## Optimization #2: Evaluator Skip Based on Technical Inventory

**Goal**: Use pre-computed `ITechnicalInventory` to skip clearly irrelevant evaluators before sending them to the LLM.

### Skip Rules

| Evaluator | Skip when |
|-----------|-----------|
| `database-patterns-coverage` | No DB deps (postgres, mysql, mongo, prisma, typeorm, etc.), no migration/ORM/seed/repository files, no DB Docker services |
| `test-patterns-coverage` | No test deps (jest, vitest, mocha, pytest, etc.), no test files (.test.*, .spec.*), no mock/fixture patterns |
| `security` | No env vars, no config files, no dependencies at all (very minimal repo only) |

### Files to Modify

1. **New: `src/shared/evaluation/inventory-skip-rules.ts`** — Skip rule logic with `computeInventorySkips(inventory) → Map<string, SkipDecision>`
2. **New: `src/shared/evaluation/inventory-skip-rules.test.ts`** — Unit tests
3. **`src/shared/evaluation/runner.ts`** — Accept `technicalInventory` in options of both `runAllEvaluators()` and `runUnifiedEvaluation()`. Compute skip decisions once before concurrency loop, apply per-evaluator
4. **`src/shared/evaluation/engine.ts`** — Pass `technicalInventory` from context to runner calls (~line 1504, ~1907, ~1122)

### Implementation Details

- `computeInventorySkips()` returns a `Map<string, SkipDecision>` where `SkipDecision = { skip: boolean; reason?: string }`
- Call it **once** before the evaluator loop, not per-evaluator
- Skipped evaluators return existing `{ skipped: true, skipReason: "..." }` pattern already supported by `EvaluatorResult`
- If `technicalInventory` is undefined (collection failed), no skips — safe fallback
- DB dep detection: match against a set of ~30 known database package names across JS/Python/Go/Ruby/Java
- Test dep detection: match against a set of ~25 known test framework names + check `fileCountsByExtension` for test file patterns
- Docker service detection: regex match service names against `/postgres|mysql|mongo|redis|mariadb/i`

## Optimization #7: Two-Phase Triage Evaluation

**Goal**: Run a single lightweight LLM call (~$0.15-0.30) to identify which evaluators are likely to find issues, then skip the rest.

### Architecture

```
File Content + Context → Triage LLM Call → ["Y","N","?"] per evaluator → Run only "Y"/"?" evaluators
```

### Files to Create/Modify

1. **New: `src/shared/evaluation/triage.ts`** — `runTriage(fileContents, evaluatorFiles, options) → TriageResult`
2. **New: `prompts/shared/triage-prompt.md`** — Compact prompt listing all 17 evaluators with 1-line descriptions, asking for Y/N/? per evaluator
3. **New: `src/shared/evaluation/triage.test.ts`** — Unit tests
4. **`src/shared/evaluation/engine.ts`** — Integrate triage before evaluator execution in both unified and independent modes
5. **`src/shared/types/evaluation.ts`** — Add `enableTriage?: boolean` to evaluation options; add triage metadata fields

### Key Design Decisions

- **Conservative by default**: "?" (uncertain) maps to "run" — only explicit "N" causes a skip
- **Safety minimum**: If triage selects < 3 evaluators, fall back to running all (triage likely miscalibrated)
- **Parsing failure fallback**: If triage response can't be parsed, run all evaluators
- **Triage runs once**: For unified mode, single call with all file content. For independent mode, single call with root file content
- **User selection override**: If user explicitly selected evaluators (`selectedEvaluators` option), triage is skipped or only filters within user's selection
- **Metadata tracking**: Triage cost, duration, and selection counts added to evaluation metadata for transparency
- **Skipped evaluator reporting**: Triage-skipped evaluators appear in results with `skipped: true, skipReason: "Triage: ..."` for auditability
- **Opt-in**: Triage is off by default (`enableTriage: false`). Enable via CLI flag `--triage` or API option

### Triage Prompt Structure (~2K tokens)

```markdown
# Evaluator Triage
Given the AGENTS.md file(s) and project context, determine which evaluators are LIKELY to find issues.
[1-line description per evaluator]
Return JSON: {"content-quality": "Y", "structure-formatting": "N", ...}
Conservative: when in doubt, return "Y" or "?".
```

### Integration in Engine

After building `projectContext` and before `runUnifiedEvaluation`/`runAllEvaluators`, insert:
1. Check `options.enableTriage`
2. Call `runTriage()` with file content + context
3. Use `triageResult.selectedEvaluators` as `selectedEvaluators` option in runner call
4. Add synthetic skipped results for triage-excluded evaluators
5. Record triage metadata in evaluation output
